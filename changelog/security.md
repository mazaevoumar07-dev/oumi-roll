# Changelog — Безопасность

Все изменения связанные с безопасностью: CORS, rate limiting, RGPD, auth.

---

## 2026-07-02

### Исправлено
- `restaurant_settings` — включён RLS, таблица была полностью открыта для anon-ключа на чтение и запись (можно было удалённо поставить приём заказов на паузу). Добавлена политика публичного чтения, запись оставлена только через service_role (миграция 006)
- `handle_new_user()` — отозван `EXECUTE` у `anon` и `authenticated`. Триггерная функция была вызываема напрямую через `/rest/v1/rpc/handle_new_user`, хотя предназначена только для триггера `on_auth_user_created` (миграция 007)

---

## 2026-05-17 — 2026-05-20

### Добавлено
- Критические риски зафиксированы в `known_risks.md` — SQL-инъекции, webhook без верификации, отсутствие rate limiting
- Срок JWT администратора — 7 дней (зафиксировано в спеке)
- Rate limiting — требование добавлено в спецификацию

### Исправлено
- 3 критические уязвимости в спеках: отсутствие таблиц, rate limiting, авторизация курьера

---

## 2026-05-31 — 2026-06-05

### Добавлено
- Авторизация через Supabase Auth — OTP по номеру телефона (вместо mock JWT/bcrypt)
- Supabase RLS — проверка роли через `supabase.auth.getUser()` на сервере

---

## 2026-06-06 — 2026-06-07

### Добавлено
- **CORS** — API-маршруты принимают запросы только с домена ресторана (`NEXT_PUBLIC_APP_URL`)
- **Rate limiting** — ограничение числа запросов на все API-маршруты (защита от перебора и DDoS)
- **RGPD ст. 17** — `DELETE /api/users/me` — клиент может удалить свой аккаунт и все персональные данные
- Страница политики конфиденциальности — обязательное требование RGPD для французского рынка

### Исправлено
- `/api/*` исключены из глобального middleware — авторизация проверяется внутри каждого маршрута отдельно
- Устаревший `config bodyParser` удалён из webhook — мог сломать верификацию подписи Stripe
- CRLF в `NEXT_PUBLIC_APP_URL` — убран через `.trim()`, приводил к некорректным HTTP-запросам
