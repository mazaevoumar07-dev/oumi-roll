# Известные риски — OUMI ROLL

> Этот файл документирует известные риски и ограничения проекта. Часть из них требует решения до запуска, часть — после первого запуска.

---

## Критические — решить до запуска

### Stripe KYC верификация
**Риск:** Stripe требует верификацию личности владельца (паспорт), подтверждение банковского счёта и описание бизнеса. Пока верификация не пройдена — Stripe держит выплаты и может заморозить аккаунт.
**Решение:** Начать верификацию в Stripe Dashboard заблаговременно, до запуска — процесс занимает несколько дней.

### RGPD — политика конфиденциальности и Cookie Notice
**Риск:** Французский закон (RGPD) требует: страницу «Politique de confidentialité», уведомление о cookies (Google Maps API использует cookies в браузере), ссылку на политику в футере. Без этого сайт нарушает RGPD с первого дня.
**Решение:** Добавить страницу `/politique-de-confidentialite` и banner с согласием на cookies до публичного запуска.

### RGPD — право на удаление данных (статья 17)
**Риск:** Клиент вправе потребовать удаление своих персональных данных. Эндпоинта `DELETE /api/users/me` нет. Нет процедуры анонимизации заказов (имя, телефон, адрес в таблице `orders`).
**Решение:** Реализовать `DELETE /api/users/me` — обнулить личные данные пользователя, анонимизировать связанные заказы (`first_name = 'Удалён'`, `phone = null`). Заказы не удалять физически — только персональные данные.

## Важные — решить в первые недели после запуска

### SMS клиенту при отмене заказа администратором не реализован
**Риск:** Кнопка «Отменить» в панели администратора инициирует возврат через Stripe, но не отправляет SMS клиенту. Если администратор отменяет заказ из-за нехватки ролла — клиент узнаёт об этом только через несколько дней, когда увидит возврат на карте.

**Правило (зафиксировано в F-11):** при отмене оплаченного заказа из-за нехватки позиции → полная отмена + автоматический SMS клиенту через Twilio:
> *«Votre commande a été annulée car un article n'est plus disponible. Vous serez remboursé sous 5 à 10 jours ouvrés. Vous pouvez passer une nouvelle commande sur notre site — l'article indisponible n'apparaîtra plus.»*

**Решение:** При вызове `POST /api/orders/:id/cancel` администратором — после успешного Stripe refund отправлять SMS на номер телефона клиента из заказа через Twilio.

### Нет уведомления администратора при новом заказе
**Риск:** Панель обновляется каждые 30 секунд. Если владелец отошёл от устройства — новый заказ может ждать минуты без реакции. Звуковой сигнал работает только если вкладка открыта и активна.
**Решение:** Отправлять SMS или push-уведомление администратору при каждом новом заказе через Twilio (на личный номер владельца). Это отдельный Twilio-запрос в webhook при создании заказа.

### Расстояние × 1.3 — неточно для Ле-Мана
**Риск:** Ле-Ман — старый город с кольцом автодорог A11. Реальное дорожное расстояние до некоторых точек превышает прямую × 1.3 на 30–50%. Клиент попадает в зону «до 3 км» (€2.50), а курьер едет 5 км.
**Решение:** Заменить на Google Maps Distance Matrix API — считает реальное расстояние по дорогам. Это дороже ($5 за 1000 запросов вместо $2.50 для Geocoding), но честнее для курьера и точнее для ценообразования.

### Мультиязычность контента меню
**Риск:** i18next переводит интерфейс, но содержимое меню (названия роллов, описания) хранится в БД на одном языке. Администратор добавит ролл по-французски — на английском и русском будет пусто или то же самое.
**Решение:** Добавить в таблицу `menu_items` поля `name_fr`, `name_en`, `name_ru`, `description_fr`, `description_en`, `description_ru`. В форме администратора — три поля для каждого языка (необязательные для EN и RU). API возвращает нужный язык по заголовку `Accept-Language`.

### Оплата прошла, но заказ не создался в системе
**Риск:** Клиент оплатил, деньги списались, но webhook `/api/payment/webhook` не отработал (сбой сервера, ошибка БД, таймаут). Клиент не получил подтверждение, заказ не попал в панель администратора.

**Как узнать:** Stripe шлёт email владельцу на каждую успешную оплату. Если в `/admin/orders` нет заказа, а email пришёл — сбой webhook. Также проверять: Stripe Dashboard → Payments.

**Что делает Stripe автоматически:** При ошибке webhook (не-200 ответ) Stripe повторяет доставку события до 72 часов с нарастающими интервалами. В большинстве случаев заказ восстанавливается сам.

**Если автоповтор не помог:** Все данные заказа хранятся в `metadata` Payment Intent (имя, телефон, адрес, состав корзины). Владелец открывает платёж в Stripe Dashboard, читает `metadata` и вручную создаёт заказ через `/admin/orders` или связывается с клиентом напрямую.

**Что видит клиент:** Страница `/confirmation` может показать ошибку вместо подтверждения. Деньги списаны, но клиент не знает что с заказом. Нужно чтобы `/confirmation` показывала нейтральное сообщение («Votre paiement a été accepté. Nous vous confirmons votre commande par SMS dans quelques minutes»), а не сырую ошибку.

**Кто отвечает:** Владелец ресторана. Автоматического алерта нет — нужно периодически сверять Stripe Dashboard с панелью заказов при подозрении на сбой.

**Решение (после запуска):** Добавить SMS-уведомление администратору в webhook при успешном создании заказа (уже описано как риск выше). Тогда отсутствие SMS при наличии email от Stripe — явный сигнал сбоя.

## Незначительные — мониторить

### Rate limiting по IP за NAT
**Риск:** Клиенты из одного офиса или через общий WiFi имеют один IP. При пиковой нагрузке (SMS-рассылка) лимит `POST /api/delivery/calculate` (10 req/min) может ложно срабатывать для разных пользователей.
**Заметка:** При малом трафике ресторана это маловероятно. Мониторить в Vercel Dashboard после запуска.

### Нет graceful redirect при истёкшей сессии
**Риск:** Access token Supabase живёт 1 час и автоматически обновляется через refresh token (middleware обновляет при каждом запросе). Refresh token живёт 7 дней — если пользователь не заходил 7+ дней, сессия истекает и он получит 401 без предупреждения.
**Заметка:** Для MVP приемлемо — стоит добавить graceful redirect на страницу входа при 401 вместо сырой ошибки.

### Отсутствует аудит-лог администратора
**Заметка:** Нет записи кто, когда и что изменил (цены, отмены, возвраты). Для одного владельца-администратора некритично, но при найме сотрудника становится важным.
