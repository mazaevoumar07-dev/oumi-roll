# Известные риски — OUMI ROLL

> Этот файл документирует известные риски и ограничения проекта. Часть из них требует решения до запуска, часть — после первого запуска.

---

## Критические — решить до запуска

### Stripe KYC верификация
**Риск:** Stripe требует верификацию личности владельца (паспорт), подтверждение банковского счёта и описание бизнеса. Пока верификация не пройдена — Stripe держит выплаты и может заморозить аккаунт.
**Решение:** Начать верификацию в Stripe Dashboard заблаговременно, до запуска — процесс занимает несколько дней.

**Статус (2026-07-14): не актуально пока Stripe остаётся в Test mode.** Сайт запущен публично, но оплаты сейчас тестовые (Stripe Test mode, решение от 2026-07-05, переподтверждено при запуске) — реальные деньги с клиентов не списываются. KYC понадобится только при переходе на Live mode.

### ~~Автодеплой Vercel деплоил мерджи в main не как Production~~ — исправлено
**Риск:** С PR #96 по PR #105 все деплои, триггерящиеся мерджем в `main`, падали на сборке с ошибкой `@supabase/ssr: Your project's URL and API key are required`. Причина: git-триггерные деплои получали `target: null` вместо `"production"`, хотя Production Branch в настройках был указан верно (`main`) и переменные окружения были на месте — но только для окружения Production, поэтому non-production сборка их не видела. Продакшн (`oumi-roll.vercel.app`) при этом оставался на версии PR #79 несколько недель.
**Решение:** Ручной деплой через `vercel --prod` (форсирует Production-таргет) сразу чинит ситуацию. Для постоянного исправления — переподключили git-интеграцию (Settings → Git → Disconnect → Connect тот же репозиторий). Подтверждено через Vercel MCP: деплой мерджа PR #106 (`d420dd2`) получил `target: production`, `state: READY` — автодеплой снова работает.

### ~~RGPD — политика конфиденциальности и Cookie Notice~~ — исправлено
**Риск:** Французский закон (RGPD) требует: страницу «Politique de confidentialité», уведомление о cookies (если используются некритичные cookies), ссылку на политику в футере. Без этого сайт нарушает RGPD с первого дня.
**Решение:** Страница `/politique-de-confidentialite` реализована (коммит `55f92b9`), ссылка в футере есть.

**Cookie banner не требуется.** Проверено (2026-07-14): сайт ставит в браузере только httpOnly cookie сессии Supabase — строго необходимую, не требующую согласия по ст. 82 loi Informatique et Libertés. Google Maps API используется исключительно на сервере (`src/app/api/delivery/calculate/route.ts`) — карта не встраивается в браузер, сторонних cookies от Google нет. Корзина хранится в React-контексте в памяти, без cookies/localStorage. Текст страницы политики про «cookies Google Maps» был неточным (Maps не используется на клиенте) — исправлен.

### ~~RGPD — право на удаление данных (статья 17)~~ — исправлено
**Риск:** Клиент вправе потребовать удаление своих персональных данных. Эндпоинта `DELETE /api/users/me` нет. Нет процедуры анонимизации заказов (имя, телефон, адрес в таблице `orders`).
**Решение:** `DELETE /api/users/me` реализован (коммит `ffb7643`) — анонимизирует связанные заказы (`first_name = 'Utilisateur supprimé'`, `phone/email/address/comment = null`, не удаляя их физически), удаляет профиль из `public.users` и учётную запись из `auth.users`. Rate limiting: 3 запроса в час на IP.

### Регистрация клиента не работает — Twilio-аккаунт в Supabase на Trial
**Риск:** Регистрация (`/connexion`, вкладка «S'inscrire») отправляет корректный запрос в Supabase (`POST /auth/v1/signup`, номер в формате `+33...`), но Supabase не может отправить SMS с кодом подтверждения — Twilio возвращает ошибку `sms_send_failed`:
```
Error sending confirmation OTP to provider: The number +336XXXXXXXX is unverified.
Trial accounts cannot send messages to unverified numbers...
More information: https://www.twilio.com/docs/errors/21608
```
Это означает, что Twilio-аккаунт, подключённый в **Supabase Dashboard → Authentication → Providers → Phone** (SMS-провайдер для верификации телефона при регистрации), — пробный (Trial). Пробный аккаунт Twilio отправляет SMS только на номера, вручную добавленные в «Verified Caller IDs» в консоли Twilio. Из-за этого **не может зарегистрироваться ни один клиент** — только заранее верифицированные номера.

Важно не путать с `TWILIO_*` в `.env.local` — это отдельная интеграция (SMS-уведомления о заказах из бэкенда), настроенная и работающая независимо. Проблема именно в SMS-провайдере телефонной аутентификации Supabase.

**Решение:** Добавить способ оплаты в Twilio-аккаунт, подключённый в Supabase Dashboard (снимает ограничение Trial). Требует доступа к Twilio Console и Supabase Dashboard — вне зоны доступа агента, делает владелец проекта.

**Подтверждено (2026-07-14):** Владелец добавил свой номер в Twilio Console → Verified Caller IDs — регистрация с этим номером прошла полностью (SMS с кодом дошло). Диагноз подтверждён: дело именно в Trial-ограничении, больше ничего не сломано.

**Статус (2026-07-14): сайт запущен публично, ограничение сознательно НЕ снято.** Реальные клиенты с неверифицированными номерами не смогут зарегистрироваться, пока владелец не добавит способ оплаты в Twilio-аккаунт, подключённый в Supabase Dashboard.

## Важные — решить в первые недели после запуска

### Twilio-аккаунт для SMS-уведомлений (`.env.local`) тоже на Trial — SMS не доставляются
**Риск:** Отдельно от Twilio в Supabase Dashboard (см. риск про регистрацию выше) — аккаунт `TWILIO_ACCOUNT_SID`/`TWILIO_AUTH_TOKEN` из `.env.local`, используемый бэкендом напрямую (уведомление администратору о новом заказе, SMS клиенту при отмене, рассылка промо), **тоже пробный (Trial)**. Проверено напрямую через Twilio API (`GET /Accounts/{sid}.json`) 2026-07-14: `"type": "Trial"`.

Проверена полная история SMS на номер `ADMIN_PHONE` (`+33602210668`) с 5 июня по 14 июля 2026 — **все без исключения** попытки провалились с ошибкой 21608 (`unverified number`), включая:
- «Nouvelle commande #1» (2026-07-02) — считалось пройденным тестом, но SMS реально не доставилось
- «Nouvelle commande #2» (2026-07-14, сквозной E2E-тест: заказ создан, `payment_status: paid`, но `notifyAdmin()` → Twilio 21608)
- Все коды OTP на этот номер за тот же период

Значит **все три** SMS-функции сейчас реально не работают ни для одного неверифицированного номера:
- Уведомление администратору о новом заказе (`/api/payment/webhook`, PR #67)
- SMS клиенту при отмене заказа (`/api/orders/:id/cancel`, PR #76) — не тестировалось напрямую в этой сессии, но использует тот же Trial-аккаунт, поэтому почти наверняка тоже проваливается для реальных номеров клиентов
- Рассылка промо-акций (`/api/admin/sms/send`, F-09)

**Решение:** Добавить способ оплаты в этот Twilio-аккаунт (Twilio Console, не Supabase Dashboard — это другой аккаунт). Либо верифицировать конкретные номера в Verified Caller IDs для локального теста.

**Статус (2026-07-14): сайт запущен публично, ограничение сознательно НЕ снято.** Код написан верно и не блокирует создание заказов (SMS падает без ошибки для клиента), просто уведомления реально не доходят ни администратору, ни клиенту. Владелец должен периодически проверять `/admin/orders` вручную, пока это не исправлено — автоматических SMS-алертов о новых заказах нет.

### ~~SMS клиенту при отмене заказа администратором не реализован~~ — реализовано (PR #76), но не доставляется
При отмене оплаченного заказа администратором `POST /api/orders/:id/cancel` автоматически отправляет SMS клиенту через Twilio. Текст и логика зафиксированы в F-11. **Код корректен**, но см. риск выше — Twilio-аккаунт на Trial, для реальных (неверифицированных) номеров клиентов SMS не доставится.

### ~~Нет уведомления администратора при новом заказе~~ — реализовано (PR #67), но не доставляется
При успешной оплате (`payment_intent.succeeded`) `POST /api/payment/webhook` отправляет SMS на `ADMIN_PHONE` через Twilio — номер заказа, тип доставки, сумма, имя и телефон клиента (`notifyAdmin()` в `route.ts`). Отправка не блокирует ответ Stripe и не роняет создание заказа при сбое SMS. **Код корректен**, но см. риск выше — сам SMS не доставляется из-за Trial-ограничения Twilio.

### Расстояние × 1.3 — неточно для Ле-Мана
**Риск:** Ле-Ман — старый город с кольцом автодорог A11. Реальное дорожное расстояние до некоторых точек превышает прямую × 1.3 на 30–50%. Клиент попадает в зону «до 3 км» (€2.50), а курьер едет 5 км.
**Решение:** Заменить на Google Maps Distance Matrix API — считает реальное расстояние по дорогам. Это дороже ($5 за 1000 запросов вместо $2.50 для Geocoding), но честнее для курьера и точнее для ценообразования.

### Мультиязычность контента меню
**Риск:** i18next переводит интерфейс, но содержимое меню (названия роллов, описания) хранится в БД на одном языке. Администратор добавит ролл по-французски — на английском и русском будет пусто или то же самое.
**Решение:** Добавить в таблицу `menu_items` поля `name_fr`, `name_en`, `name_ru`, `description_fr`, `description_en`, `description_ru`. В форме администратора — три поля для каждого языка (необязательные для EN и RU). API возвращает нужный язык по заголовку `Accept-Language`.

### Оплата прошла, но заказ не создался в системе
**Риск:** Клиент оплатил, деньги списались, но webhook `/api/payment/webhook` не отработал (сбой сервера, ошибка БД, таймаут). Клиент не получил подтверждение, заказ не попал в панель администратора.

**Как узнать:** Stripe шлёт email владельцу на каждую успешную оплату. Если в `/admin/orders` нет заказа, а email пришёл — сбой webhook. Также проверять: Stripe Dashboard → Payments.

**Что делает Stripe автоматически:** При ошибке webhook (не-200 ответ) Stripe повторяет доставку события до 72 часов с нарастающими интервалами. В большинстве случаев заказ восстанавливается сам.

**Если автоповтор не помог:** Все данные заказа хранятся в `metadata` Payment Intent (имя, телефон, адрес, состав корзины). Владелец открывает платёж в Stripe Dashboard, читает `metadata` и вручную создаёт заказ через `/admin/orders` или связывается с клиентом напрямую.

**Что видит клиент:** Страница `/confirmation` может показать ошибку вместо подтверждения. Деньги списаны, но клиент не знает что с заказом. Нужно чтобы `/confirmation` показывала нейтральное сообщение («Votre paiement a été accepté. Nous vous confirmons votre commande par SMS dans quelques minutes»), а не сырую ошибку.

**Кто отвечает:** Владелец ресторана. Автоматического алерта нет — нужно периодически сверять Stripe Dashboard с панелью заказов при подозрении на сбой.

**Решение (после запуска):** Добавить SMS-уведомление администратору в webhook при успешном создании заказа (уже описано как риск выше). Тогда отсутствие SMS при наличии email от Stripe — явный сигнал сбоя.

## Незначительные — мониторить

### Rate limiting по IP за NAT
**Риск:** Клиенты из одного офиса или через общий WiFi имеют один IP. При пиковой нагрузке (SMS-рассылка) лимит `POST /api/delivery/calculate` (10 req/min) может ложно срабатывать для разных пользователей.
**Заметка:** При малом трафике ресторана это маловероятно. Мониторить в Vercel Dashboard после запуска.

### Нет graceful redirect при истёкшей сессии
**Риск:** Access token Supabase живёт 1 час и автоматически обновляется через refresh token (middleware обновляет при каждом запросе). Refresh token живёт 7 дней — если пользователь не заходил 7+ дней, сессия истекает и он получит 401 без предупреждения.
**Заметка:** Для MVP приемлемо — стоит добавить graceful redirect на страницу входа при 401 вместо сырой ошибки.

### Отсутствует аудит-лог администратора
**Заметка:** Нет записи кто, когда и что изменил (цены, отмены, возвраты). Для одного владельца-администратора некритично, но при найме сотрудника становится важным.
